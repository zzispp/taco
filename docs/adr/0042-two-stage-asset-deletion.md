# Use two-stage asset deletion

Deleting a Managed File or Logical Folder moves the asset and its subtree into the recycle bin while retaining metadata and stored content for restoration. Provider content is removed only during explicit Permanent Deletion, and Business References block both deletion stages. After a replacement avatar is successfully bound to a user, the previous Avatar Asset loses its Business Reference and automatically enters the recycle bin; the user reference update, prior-asset status transition, usage recalculation, and audit outbox append commit in one PostgreSQL transaction. The old asset is never removed before the new reference succeeds, and an API failure cannot report a committed replacement. Trashed Assets become eligible for scheduled Permanent Deletion after a default 30-day Trash Retention Period. The Scheduler job exclusively owns the retention and batch parameters and snapshots them for each execution; File Management owns eligibility and deletion behavior, and no duplicate `sys_config` value exists. A migration seeds the cleanup job once for fresh and upgraded installations. This reduces shared-library data loss and gives object-storage cleanup a visible lifecycle instead of pretending database and Provider deletion form one atomic transaction.
