# Use resumable multipart uploads

Managed Files are uploaded through resumable Upload Sessions rather than whole-file requests. The server assigns a 16 MiB Upload Part size when a session is created, except that a Provider may require a larger value; every non-final part matches the assigned size, and the value cannot change during the session. Every part carries a server-verified SHA-256 digest: retrying the same part number and digest is idempotent, while different content conflicts. Completion verifies the full-file SHA-256 Content Digest and stores it as a read-only File Property. Clients can query accepted parts and resume, and only a successfully completed and fully validated session atomically creates the Managed File metadata. If a verified Stored Object with the declared digest and size exists inside the caller's Content Reuse Scope, session creation skips part transfer and immediately creates or idempotently returns the target Managed File; a match outside that scope requires Proof of Possession. A session expires after a default seven-day Upload Session Inactivity Window, refreshed by every successfully accepted part; File Management runtime configuration exclusively owns that value. The same configuration owns a default 10 GiB Managed File Size Limit, enforced when the session is created and again when it completes. Local storage assembles temporary parts into an opaque object, while future OSS and MinIO adapters may map the same application contract to native multipart uploads.
